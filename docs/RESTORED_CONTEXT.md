# Restored Context

## Что это за workspace

`codex-workspace` был выделен внутри репозитория `Olympus-Athena-Antigravity` как отдельная изолированная рабочая зона под новый проект:

- название: `Content Autopilot`;
- тип: личный инструмент автоконтент-маркетинга;
- цель: автоматически вести несколько продуктов через цикл `продукт -> контент-план -> статья -> блог -> VK`;
- позиционирование: не SaaS, а персональный рабочий инструмент владельца продуктов.

Это подтверждается файлами:

- `README.md`
- `docs/product-brief.md`
- `docs/architecture.md`
- `docs/MEMORY.md`


## Главные продуктовые решения, которые уже были приняты

- Не смешивать новый проект со старой системой в корне репозитория.
- Работать только внутри `codex-workspace`.
- Делать не универсальную платформу, а практичный личный инструмент.
- Максимально убирать ручной труд.
- Оставлять человеку только аварийное вмешательство:
  - исправить ошибку;
  - перегенерировать неудачный материал;
  - поменять настройки продукта, стиля или публикации.
- Базовый KPI проекта: стабильный выпуск полезного контента при минимуме ручных действий.


## Технические решения, которые уже были приняты

- Backend: `FastAPI`
- Frontend: `Next.js`
- Database: `PostgreSQL`
- Queue / background: `Redis + Celery`
- Infra: `Docker Compose`
- LLM-провайдер в планах ядра: `OpenRouter`

Дополнительные ограничения из памяти проекта:

- `Supabase` и `n8n` не считаются обязательной частью ядра.
- `.env` и временные локальные файлы нельзя коммитить.
- `master-ai` не трогать без отдельной задачи.

## Что видно по последнему зафиксированному состоянию

Последний коммит, который создал эту рабочую зону:

- `ef5cd0e` — `Add athena content autopilot workspace`

Это означает, что workspace добавлялся как отдельный крупный этап, а не как случайный набор файлов. Судя по структуре, работа шла в таком порядке:

1. Выделить отдельную рабочую область.
2. Зафиксировать продуктовую рамку и ограничения.
3. Поднять инфраструктурный каркас.
4. Зафиксировать data model.
5. Реализовать первую волну API.
6. Оставить handover и roadmap для следующей итерации.

## Практический вывод

Проект находился в состоянии "фундамент уже собран, следующая задача — переход от каркаса к первому рабочему контентному циклу".

Если продолжать в том же направлении без лишнего риска, логичный порядок такой:

1. Завести первый реальный `Product`.
2. Завести `BrandProfile`.
3. Реализовать `ContentPlan` CRUD.
4. Добавить bootstrap/seed сценарий для первого продукта.
5. После этого строить генерацию и автопилот поверх уже живых сущностей.


## Что стоит считать рабочей гипотезой, а не фактом

Ниже выводы, которые восстановлены по косвенным признакам, а не напрямую записаны в одном месте:

- работа велась итеративно от архитектуры к первому production-like scaffold;
- следующий фокус почти наверняка должен был быть на `ContentPlan` и bootstrap первого продукта;
- идея была сначала добиться рабочего цикла на одном продукте, а уже потом расширять автоматизацию и публикацию.

Это очень сильная гипотеза, потому что она одновременно подтверждается:

- handover;
- TODO;
- project state;
- roadmap;
- текущим набором реализованных моделей и endpoints.
